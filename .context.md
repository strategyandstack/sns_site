# AI Working Context

## Project Summary
*   Vanilla stack marketing site focused on "Sales Architecture" blueprints.
*   Data-driven rendering: Data (`data.js`) → Components (`components.js`) → Orchestration (`main.js`) → DOM.
*   Zero framework overhead; high performance through minimal dependencies.
*   GSAP used for premium visual interactions and complex scroll effects.
*   Lucide icons handled via UMD CDN and programmatic creation.
*   Spanish (`es`) version exists alongside English, sharing same logic and components.
*   Strict security headers and CSP via `_headers` file.
*   Self-hosted fonts (Inter) preloaded for zero CLS.
*   Bilingual SEO setup with `hreflang` and AI-optimized `robots.txt`.
*   Deployment targets static platforms like Cloudflare Pages or Netlify.

## Ground Rules
*   **Prefer modifying `data.js`**: Content, prices, features, and config live here.
*   **Prefer modifying `components.js`**: This is where HTML structure and layout logic reside.
*   **Prefer modifying `main.js`**: Use for hydration logic, event listeners, and animation setup.
*   **No Inline Scripts**: Breaking CSP is strictly forbidden. All logic belongs in modules.
*   **No New Frameworks**: Keep the stack pure vanilla JS/CSS.
*   **Modular Imports**: Use `?v=N` versioning on module imports in `main.js` if updating across versions.

## Where to Change Things
*   **Marketing Copy/Prices**: `data.js`
*   **HTML Structure/UI Layout**: `components.js`
*   **Animation Logic/GSAP Tweaks**: `main.js`
*   **Global Styles/CSS Tokens**: `style.css`
*   **Security Policies/Caching**: `_headers`
*   **SEO Metadata/Schema**: `index.html`

## Style Conventions
*   **Naming**: CamelCase for functions, kebab-case for CSS classes/IDs.
*   **Components**: Must be pure functions returning template literals.
*   **Hydration**: Sections should be mounted via `innerHTML` or `insertAdjacentHTML` in `main.js`.
*   **Icons**: Reference Lucide icons by name in data; call `lucide.createIcons()` after DOM updates.

## PR Checklist Before Final Answer
- Identify all files to change (Data, Component, or Logic level).
- Explain why each change belongs there (e.g., "Content change in data.js").
- Keep changes minimal and focused.
- Ensure no CSP violations in the proposed code.
- Validate JSON-LD syntax if modifying schema.
- Confirm ES module imports are correct and typed correctly.
- Confirm GSAP init is called once per element or timeline.
- Suggest a local test plan: "Run local server, verify refresh, check mobile responsiveness."
